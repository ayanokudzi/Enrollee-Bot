# Design-Doc: Чат бот "Enrollment 2026" / RAG-система
**Проектная команда:**
* **Бизнес-заказчик:** Факультет компьютерных наук НИУ ВШЭ (Приемная комиссия).
* **Разработчик чат-бота:** Ковязин Алексей Дмитриевич.
## 1. Контекст проекта
### 1.1 Бизнес-задача
* у абитуриентов много вопросов и мало ответов, разобраться в многочисленных справочниках и документах будет далеко не просто. приемная комиссия может быть перегружена в связи с большим потоком поступающих.
* создать обычного бота с кнопками не получится - количество вопросов от абитуриентов неисчислимо велико, поэтому используем rag систему. также выбор системы обусловлен динамичностью данных (достаточно обновить текстовый файл в базе знаний) и точностью ответов (система идет строго по официальным документам).
* успех проявляется в облегчении работы приемной коммиссии, сокращении времени ответа.
### 1.2 Целевая аудитория и пользователи
* Чат-ботом будут пользоваться все кому интересно поступление - от абитуриентов до их родителей.
* Взаимодействие с ботом просиходит в мессенджере Telegram. Выбор обучен тем, что это основной канал коммуникации абитуриентов и студентов ФКН. Абитуриент задает вопрос в свободной форме, а бот трансформирует сложный язык официальных документов в простой и понятный ответ.
* Ожидаемая нагрузка 5000-10000 пользователей за сезон. Пиковая нагрузка с 12:00 по 18:00 (UTC+3).
### 1.3 Ограничения и допущения
* Бот не будет поддерживать пустое общение, будут обсуждаться только вопросы с поступлением. Также бот не будет давать прогнозов по проходным баллам, только статистика прошлых лет.
* Ответы бота не являются официальной офертой.
## 2. Архитектура решения
### 2.1 Общая схема
* Telegram Interface (Принимает сообщение) -> Backend (Python) (чистит запрос и отправляет в RAG движок) -> Retrieval (Стучимся в векторную БД) -> LLM Prompt (В коде лежит шаблон промпта. В него вставляется вопрос + найденный текст.) -> Generation (LLM дает ответ) -> Post-processing (Бот добавляет ссылку на источник и отправляет пользователю)
### 2.2 Хранилище знаний / документооборот
* Источниками являются PDF/тексты с официального сайта ФКН.
* Векторное представление хранится в базе FAISS. Исходные тексты лежат в папке data/.
* Обновление ручное.
### 2.3 Retrieval + Generation
* Retrieval: Модель sentence-transformers
* Generation: Использование API (например, GPT-3.5/4)
### 2.4 Интеграции и интерфейсы
* Интерфейс: Telegram Bot API
* Все диалоги сохраняются в текстовый лог
* Кнопки быстрого доступа к популярным разделам
### 2.5 Инфраструктура и развертывание
* Развертывание происходит через виртуальный сервер (хостер vdsina). Имеется готовый сервер с CPU 1 core, 2GB RAM.
## 3. Данные и качество знаний
**Пометка о безопасности:** Используемые документы (правила приема, описания программ) являются публичными и не содержат персональных данных. Это позволяет использовать внешние API (например, OpenAI) без риска нарушения конфиденциальности или законодательства.
### 3.1 Сбор и предобработка данных
* Источники: официальный сайт ФКН ВШЭ, PDF регламенты, справочники FAQ от приемной коммиссии.
* Предобработка заключается в переводе PDF и HTML страниц в текстовый формат.
* Для того чтобы поиск находил абзац а не весь документ целиком, разобьем текст на небольшие смысловые фрагменты по 500-800 символов с пересечением в 10%.
### 3.2 Векторизация и индексирование
* Модель Embedding: Будет использована модель семейства Sentence-Transformers
* Алгоритм: FAISS
* Так как объем данных ФКН относительно небольшой, выбрана полная пересборка индекса при каждом обновлении базы знаний. Это гарантирует отсутствие ошибок в индексации и простую поддержку кода.
### 3.3 Метрики качества знаний
* База данных обязана включать важные темы (БВИ, перечень ЕГЭ, сроки).
* Синхронизация данных в течение суток после публикации изменений на сайте ФКН. Для абитуриентов важна актуальность информации.
## 4. Модель и генерация
### 4.1 Выбор LLM и промптинг
* Model GPT-4. Хорошо справляется с инструкциями и владеет русским языком.
* В промпт всегда передается найденный кусок текста из базы знаний. Инструкция: "Отвечай только по тексту. Не используй внешние знания и будь вежлив. Если нет информации скажи об этом напрямую."
* Ограничения - лимит до 500 токенов.
### 4.2 Контроль качества ответов
* Самым главным показателем является accuracy. Все ответы, где модель выразила неуверенность или получила негативный фидбек, уходят на ручную проверку.
* Если релевантность найденного текста в базе знаний ниже порога, бот не генерирует ответ, а выдает Fallback.
### 4.3 Обучение/дообучение
* Полноценное дообучение модели не планируется.
## 5. UX / пользовательский опыт
### 5.1 Сценарии взаимодействия
* При нажатии /start бот выдает привествтвие где сообщает пользователю что могут быть неточности.
* RAG ответ - выдает ответ (строго привязанный к документам) а также ссылку на источник.
* Поддержка "памяти" на 2-3 реплики.
* Если случается fallback то бот пишет сообщение: "К сожалению, в регламентах нет точного ответа. Обратитесь на почту priem-fcs@hse.ru"
### 5.2 Диалоговая логика
* Поддерживается multi-turn диалог. Это позволяет пользователю задавать уточняющие вопросы, не повторяя тему.
* Временная история хранится в оперативной памяти с TTL 1 час.
### 5.3 Метрики UX
* Время генерации ответа примерно 5 секунд. Чтобы пользователь не скучал, бот сразу отправляет статус "печатает..." в чат.
* CSAT - под каждым ответом стоит кнопка лайка и дизлайка.
## 6. Безопасность, соответствие и этика
* Бот не запрашивает и не сохраняет ФИО, паспортные данные или СНИЛС. В интерфейсе будет закреплено предупреждение: "Пожалуйста, не вводите свои персональные данные в чат". Проект соответствует ФЗ-152 "О персональных данных", так как не осуществляет их обработку.
* Промпт блокирует попытки использовать бота не по назначению (написание эссе, решение задач, политические дискуссии).
* Бот не имитирует живого человека, чтобы не вводить абитуриентов в заблуждение. Каждый ответ, содержащий факты (баллы, даты), сопровождается ссылкой на официальный документ.
## 8. Риски и допущения
* Бот может ошибаться и давать недостоверную информацию. Например ложные баллы либо даты. (Галлюцинации модели)
* Зависимость от зарубежных сервисов (OpenAI)
* Недостаток знаний: база может не содержать ответов на специфические частные случаи.
## 10. Приложения
* RAG (Retrieval-Augmented Generation) - метод обогащения запроса к LLM данными из внешних документов.
* Эмбеддинг - числовое векторное представление смысла текста.
* Галлюцинация - генерация моделью фактологически неверной, но правдоподобной информации.
* FAISS (Facebook AI Similarity Search) - библиотека для быстрого поиска ближайших векторов в больших базах данных
* Check list
[ ] Content Check: В базе отсутствуют документы за 2024/2025 учебный год.

[ ] Technical Check: Время генерации ответа на пиковой нагрузке не превышает 7 секунд.

[ ] Safety Check: Бот корректно отклоняет запросы на отвлеченные темы (политика, код, эссе).

[ ] Fallback Check: Кнопка перехода на почту приемной комиссии корректно отображается при отсутствии ответа.

[ ] UX Check: Все ссылки в ответах бота являются рабочими и ведут на актуальные разделы сайта
